# Spec: Plane agent service account

**Status:** active
**Plane:** ec0b624a-d3cd-4f0f-9140-ecd074574532
**Created:** 2026-06-30

## Problem

Комментарии SDD-агентов (Supercode MCP, Cloud orchestrator, `agent-clarification.sh`) публикуются от имени владельца `PLANE_API_KEY` — обычно это личный аккаунт человека. В треде work item нельзя отличить сообщение агента от сообщения оператора. Нужна отдельная учётная запись Plane для агентов и единая политика ключей: **админ-ключ** для orchestrator/CI (листинг, смена state), **agent-ключ** для комментариев — чтобы в UI Plane автор отображался как сервисный пользователь (например «AtlasQuant Agent»), а роль SDD — в тексте комментария (`[Architect]`, `[Implement]` и т.д., см. spec #8).

## Acceptance Criteria

- [x] AC-1: `docs/agent-pipeline/plane-agent-account.md` описывает создание пользователя Plane (email, display name «AtlasQuant Agent», роль Member в workspace/project), выпуск API key, заполнение `PLANE_AGENT_API_KEY` в `.orchestrator/.env` и `.supercode/workflows/atlasquant/.env`; явно разделяет `PLANE_API_KEY` (admin/orchestrator) vs `PLANE_AGENT_API_KEY` (comments).
- [x] AC-2: `bash .supercode/workflows/atlasquant/scripts/verify-plane-agent.sh` GET `/api/v1/users/me/` с `PLANE_AGENT_API_KEY` (fallback `PLANE_API_KEY`); exit 0 и stdout содержит `display_name` при успехе; exit 1 при отсутствии env или HTTP ≠ 200; опция `--expect "AtlasQuant Agent"` exit 1 если display_name не совпадает.
- [x] AC-3: `.orchestrator/src/config.ts` экспортирует `resolvePlaneAgentApiKey()` — `PLANE_AGENT_API_KEY ?? PLANE_API_KEY`; unit-тест проверяет fallback и приоритет agent-ключа.
- [x] AC-4: `.orchestrator/src/poll-plane.ts` и `.orchestrator/src/start-agent.ts` используют `PLANE_API_KEY` для `listWorkItems` / `getWorkItem` / `updateWorkItemState`, и отдельный `PlaneClient` с `resolvePlaneAgentApiKey()` для `addComment`.
- [x] AC-5: `.supercode/workflows/atlasquant/scripts/plane-mcp-server.sh` экспортирует в MCP `PLANE_API_KEY=${PLANE_AGENT_API_KEY}` когда agent-ключ задан (иначе прежний `PLANE_API_KEY`); `plane-mcp-context.sh` (base) документирует dual-key policy.
- [x] AC-6: `.supercode/workflows/atlasquant/scripts/update-plane-state.sh` и `agent-clarification.sh` POST comment через agent-ключ; PATCH state через `PLANE_API_KEY` (admin).
- [x] AC-7: `.orchestrator/.env.example` и `.supercode/workflows/atlasquant/.env.example` содержат комментарий с ссылкой на `plane-agent-account.md` и ожидаемый display name.
- [x] AC-8: `npm test` в `.orchestrator` exit 0; `bin/ci` exit 0.

## Out of Scope

- Автоматическое создание пользователя Plane через API (только runbook для человека)
- Изменения Rails-приложения AtlasQuant
- Отдельный API key на каждую роль SDD (Architect, Implement, …) — роль в тексте комментария
- GitHub Actions secrets для agent-ключа (локальный/Supercode/orchestrator only)
- Парсинг display name из Plane для динамической подписи

## Constraints

- MVP boundaries: только agent pipeline / orchestrator / docs
- Секреты только через `.env`; не коммитить ключи
- Обратная совместимость: без `PLANE_AGENT_API_KEY` поведение как раньше (единый ключ)
- Минимальный diff: не рефакторить unrelated orchestrator logic
